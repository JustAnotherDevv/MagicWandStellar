markdown"><p>Authorization is the process of judging which operations &quot;should&quot; or &quot;should not&quot; be allowed to occur; it is about judging <em>permission</em>.</p>
<p>Authorization differs from <em>authentication</em>, which is the narrower problem of judging whether a person &quot;is who they say they are&quot;, or whether a message claiming to come from a person &quot;really&quot; came from them.</p>
<p>Authorization often uses cryptographic authentication (via signatures) to support its judgments, but is a broader, more general process.</p>
<p>View the <a class="" href="/docs/build/guides/auth/contract-authorization">authorization starter guide</a> to learn more about smart contract authorization on Stellar.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="soroban-authorization-framework">Soroban Authorization Framework<a href="#soroban-authorization-framework" class="hash-link" aria-label="Direct link to Soroban Authorization Framework" title="Direct link to Soroban Authorization Framework" translate="no">​</a></h2>
<p>Soroban aims to provide a light-weight, but flexible and extensible framework that allows contracts to implement arbitrarily complex authorization rules, while providing built-in implementation for some common tasks (such as replay prevention). The framework consists of the following components:</p>
<ul>
<li class="">Contract-specific authorization - custom authorization rules implemented by contracts using the private contract storage and abstract accounts.</li>
<li class="">Account abstraction - allows users to customize their authentication rules and define universal authorization policies via contract account. This includes the built-in support for Stellar accounts.</li>
<li class="">Host-based authorization library - ensures integrity between contract accounts and regular contracts. Also defines the structured signature payload format, ensures replay prevention and takes care of providing the correct signature contexts.</li>
</ul>
<p>Soroban host also provides some cryptographic functions (signature verification, hashing) which may be useful for contract account implementations.</p>
<p>Contracts that use Soroban authorization framework are interoperable with each other. Also it is easier for the client applications to write generic code for interaction with Soroban authorization framework. For example, wallets can implement a generalized way to present and sign Soroban payloads.</p>
<p>We realize that it&#x27;s not possible to cover each and every case, but we hope that the vast majority of the contracts can operate within the framework and thus contribute to building a more cohesive ecosystem. Custom authorization frameworks are still possible to implement, but are not encouraged (unless there are no alternatives).</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="contract-specific-authorization">Contract-Specific Authorization<a href="#contract-specific-authorization" class="hash-link" aria-label="Direct link to Contract-Specific Authorization" title="Direct link to Contract-Specific Authorization" translate="no">​</a></h3>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="contract-storage">Contract storage<a href="#contract-storage" class="hash-link" aria-label="Direct link to Contract storage" title="Direct link to Contract storage" translate="no">​</a></h4>
<p>Contracts have an exclusive read and write access to their <a class="" href="/docs/build/smart-contracts/getting-started/storing-data">storage</a> in the ledger. This allows contracts to safely control and manage user access to their data. For example, a token contract may ensure that only the administrator can mint more of the token by storing the administrator identity in its storage. Similarly, it can make sure that only an owner of the balance may transfer that balance.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="address"><code>Address</code><a href="#address" class="hash-link" aria-label="Direct link to address" title="Direct link to address" translate="no">​</a></h4>
<p>The storage-based approach described in the previous section requires a way to represent the user identities and authenticate them. <code>Address</code> type is a host-managed type that performs these functions.</p>
<p>From the contract perspective <code>Address</code> is an opaque identifier type. The contract logic doesn&#x27;t need to depend on the internal representation of the <code>Address</code> (see <a href="#account-abstraction" class="">Account Abstraction</a> section below for more details).</p>
<p><code>Address</code> type has two similar methods in Soroban SDK: <code>require_auth</code> and <code>require_auth_for_args</code> (these methods call the respective Soroban host function). The only difference between the functions is the ability to customize the invocation arguments. See <a class="" href="/docs/build/smart-contracts/example-contracts/auth">auth example</a> that demonstrates how to use these functions.</p>
<p>Both functions ensure that the <code>Address</code> has authorized the call of the current function within the current context (where context is defined by <code>require_auth</code> calls in the current call stack; see more formal definition in the <a href="#require_auth-implementation-details" class="">section below</a>). The authentication rules for this authorization are defined by the <code>Address</code> and are enforced by the Soroban host. Replay protection is also implemented in the host, i.e., there is normally no need for a contract to manage its own nonces.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="authorizing-sub-contract-calls">Authorizing Sub-contract Calls<a href="#authorizing-sub-contract-calls" class="hash-link" aria-label="Direct link to Authorizing Sub-contract Calls" title="Direct link to Authorizing Sub-contract Calls" translate="no">​</a></h4>
<p>One of the key features of Soroban Authorization Framework is the ability to easily make authorized sub-contract calls. For example, it is possible for a contract to call <code>require_auth</code> for an <code>Address</code> and then call <code>token.xfer</code> authorized for the same <code>Address</code> (see <a class="" href="/docs/build/smart-contracts/example-contracts/timelock">timelock example</a> that demonstrates this pattern).</p>
<p>Contracts don&#x27;t need to do anything special to benefit from this feature. Just calling a sub-contract that calls <code>require_auth</code> will ensure that the sub-contract call has been properly authorized.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="when-to-require_auth">When to <code>require_auth</code><a href="#when-to-require_auth" class="hash-link" aria-label="Direct link to when-to-require_auth" title="Direct link to when-to-require_auth" translate="no">​</a></h4>
<p>The main authorization-related decision a contract writer needs to make for any given <code>Address</code> is whether they need to call <code>require_auth</code> for it. While the decision needs to be made on case-by-case basis, here are some rules of thumb:</p>
<ul>
<li class="">If the access to the <code>Address</code> data in this contract is read-only, then <code>require_auth</code> is probably not needed.</li>
<li class="">If the <code>Address</code> data in this contract is being modified in a way that&#x27;s not strictly beneficial to the user, then <code>require_auth</code> is probably needed (e.g. reducing the user&#x27;s token balance needs to be authorized, while increasing it doesn&#x27;t need to be authorized)</li>
<li class="">If a contract calls another contract that will call <code>require_auth</code> for the <code>Address</code> (e.g. <code>token.xfer</code>), then adding <code>require_auth</code> in the caller would ensure that the authorization for the inner call can&#x27;t be reused outside of your contract. For example, if you want to do something positive for the user, but only when they have transferred some token to your contract, then the contract call itself should <code>require_auth</code>.</li>
</ul>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="authorizing-multiple-addresses">Authorizing Multiple <code>Address</code>es<a href="#authorizing-multiple-addresses" class="hash-link" aria-label="Direct link to authorizing-multiple-addresses" title="Direct link to authorizing-multiple-addresses" translate="no">​</a></h4>
<p>There is no explicit restriction on how many <code>Address</code> entities the contract uses and how many <code>Address</code>es have <code>require_auth</code> called. That means that it is possible to authorize a contract call on behalf of multiple users, which may even have different authorization contexts (customized via arguments in <code>require_auth_for_args</code>). <a class="" href="/docs/build/smart-contracts/example-contracts/atomic-swap">Atomic swap</a> is an example that deals with authorization of two <code>Address</code>es.</p>
<p>Note though, that contracts that deal with multiple authorized <code>Address</code>es need a bit more complex support on the client side (to collect and attach the proper signatures).</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="account-abstraction">Account Abstraction<a href="#account-abstraction" class="hash-link" aria-label="Direct link to Account Abstraction" title="Direct link to Account Abstraction" translate="no">​</a></h3>
<p>Account abstraction is a way to decouple the authentication logic from the contract-specific authorization rules. The <code>Address</code> defined above is in fact an identifier of an &#x27;abstract&#x27; account. That is, the contracts know the <code>Address</code> and can require authorization from it, but they don&#x27;t know how exactly it is implemented.</p>
<p>For example, imagine a token contract. Its responsibilities are to manage the balances of multiple users (transfer, mint, burn etc.). There is really nothing about these responsibilities that has anything to do with <em>how exactly</em> the user authorized the balance-modifying transaction. The users may want to use some hardware key that supports a new generation of crypto algorithms(which don&#x27;t even have to exist today) or they may want to have bespoke multisig scheme and none of this really has anything to do with the token logic.</p>
<p>Account abstraction provides a convenient extension point for every contract that uses <code>Address</code> for authorization. It doesn&#x27;t solve all the issues automatically - client-side tooling may still need to be adapted to support different authentication schemes or different wallets. But the on-chain state doesn&#x27;t need to be modified and modifying the on-chain state is a much harder problem.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="types-of-account-implementations">Types of Account Implementations<a href="#types-of-account-implementations" class="hash-link" aria-label="Direct link to Types of Account Implementations" title="Direct link to Types of Account Implementations" translate="no">​</a></h4>
<p>Conceptually, every abstract account is a special contract that defines authentication rules and potentially some additional account-specific authorization policies. However, for the sake of optimization and integration with the existing Stellar accounts, Soroban supports 4 different kinds of the account implementations.</p>
<p>Below are the general descriptions of these implementations. See the transaction <a class="" href="/docs/learn/fundamentals/contract-development/contract-interactions/stellar-transaction">guide</a> for the concrete information of how different accounts are represented.</p>
<h5 class="anchor anchorTargetStickyNavbar_Vzrq" id="stellar-account">Stellar Account<a href="#stellar-account" class="hash-link" aria-label="Direct link to Stellar Account" title="Direct link to Stellar Account" translate="no">​</a></h5>
<p>Corresponds to <code>Address::Account</code>.</p>
<p>This is a special, built-in &#x27;account contract&#x27; that handles all the Stellar accounts. It is not a real contract and doesn&#x27;t need to be deployed.</p>
<p>This supports the Stellar multisig with medium threshold. See Stellar <a class="" href="/docs/learn/fundamentals/transactions/signatures-multisig">documentation</a> for more details on multisig and thresholds.</p>
<h5 class="anchor anchorTargetStickyNavbar_Vzrq" id="transaction-invoker">Transaction Invoker<a href="#transaction-invoker" class="hash-link" aria-label="Direct link to Transaction Invoker" title="Direct link to Transaction Invoker" translate="no">​</a></h5>
<p>Corresponds to <code>Address::Account</code>.</p>
<p>This is also a Stellar account, but its signature is inferred from the source account of the Stellar transaction (or operation, if it has one).</p>
<p>This is purely an optimization of the <a href="#stellar-account" class="">Stellar Account</a> that can skip one signature in case the transaction source account also authorizes the contract invocation.</p>
<h5 class="anchor anchorTargetStickyNavbar_Vzrq" id="contract-invoker">Contract Invoker<a href="#contract-invoker" class="hash-link" aria-label="Direct link to Contract Invoker" title="Direct link to Contract Invoker" translate="no">​</a></h5>
<p>Corresponds to <code>Address::Contract</code>.</p>
<p>This is a special case of an &#x27;account&#x27; that may appear only when a contract calls another contract. We consider that since the contract makes a call, then it must be authorizing it (otherwise, it shouldn&#x27;t have made that call). Hence all the <code>require_auth</code> calls made on behalf of the <strong>direct</strong> invoker contract <code>Address</code> are considered to be authorized (but not any calls on behalf of the contract deeper down the stack).</p>
<h5 class="anchor anchorTargetStickyNavbar_Vzrq" id="contract-account">Contract Account<a href="#contract-account" class="hash-link" aria-label="Direct link to Contract Account" title="Direct link to Contract Account" translate="no">​</a></h5>
<p>Corresponds to <code>Address::Contract</code>.</p>
<p>This is the extension point of account abstraction. A contract that implements the <code>CustomAccountInterface</code> and <code>__check_auth</code> becomes a contract account. If any contract calls <code>require_auth</code> for the <code>Address</code> of this contract, the Soroban host will call <code>__check_auth</code> with the corresponding arguments.</p>
<p><code>__check_auth</code> gets a signature payload, a list of signatures (in any user-defined format) and a list of the contract invocations that are being authorized by these signatures. Its responsibility is to perform the authentication via verifying the signatures and also (optionally) to apply a custom authorization policy. For example, a signature weight system similar to Stellar can be implemented, but it also can have customizable rules for the weights, e.g. to allow spending more than X units of token Y only given signature weight Z.</p>
<p>Contract accounts can also be treated as a custodial wallet. It holds the user&#x27;s funds (token balances, NFTs etc.) and provides the user(s) with ways to authorize operations on these funds. Nothing prevents contract accounts from authorizing operations unrelated to balances; for example, they can perform administrative functions for tokens (contract accounts define what to do when <code>require_auth</code> is called).</p>
<p>For the exact interface and more details, see the <a class="" href="/docs/build/smart-contracts/example-contracts/simple-account">Simple Account example</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="secp256r1-passkeys-and-contract-accounts">Secp256r1, passkeys and contract accounts<a href="#secp256r1-passkeys-and-contract-accounts" class="hash-link" aria-label="Direct link to Secp256r1, passkeys and contract accounts" title="Direct link to Secp256r1, passkeys and contract accounts" translate="no">​</a></h3>
<p>After a successful public validator vote to upgrade Stellar&#x27;s Mainnet to Protocol 21, the secp256r1 signature scheme was enabled for smart contract transactions. This allows developers to implement passkeys to sign transactions instead of using secret keys or seed phrases. For guidance, see the <a class="" href="/docs/build/guides/contract-accounts/smart-wallets">passkey wallet guide</a>.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="advanced-concepts">Advanced Concepts<a href="#advanced-concepts" class="hash-link" aria-label="Direct link to Advanced Concepts" title="Direct link to Advanced Concepts" translate="no">​</a></h3>
<p>Most of the contracts shouldn&#x27;t need the concepts described in this section. Refer to this when developing complex contracts that deal with deep contract call trees and/or multiple <code>Address</code>es.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="require_auth-implementation-details"><code>require_auth</code> implementation details<a href="#require_auth-implementation-details" class="hash-link" aria-label="Direct link to require_auth-implementation-details" title="Direct link to require_auth-implementation-details" translate="no">​</a></h4>
<p>When a Soroban transaction is executed on-chain, the host collects a list of <code>SorobanAuthorizationEntry</code> entries from the transaction (<a href="https://github.com/stellar/stellar-xdr/blob/e372df9f677961aac04c5a4cc80a3667f310b29f/Stellar-transaction.x#L570" target="_blank" rel="noopener noreferrer" class="">XDR</a>). These entries contain signed authorizer credentials and authorized invocation trees. The host uses these entries to verify authorization during the contract execution.</p>
<p>Every time <code>require_auth</code>/<code>require_auth_for_args</code> host function is called for non-contract-invoker account, the following steps happen:</p>
<ul>
<li class="">Find an authorized invocation tree that matches the <code>require_auth</code> call. The matching process is pretty involved and is described in the section below.</li>
<li class="">If authentication hasn&#x27;t happened for this tree yet, then perform it:<!-- -->
<ul>
<li class="">Verify signature expiration. Expired signatures are not valid.</li>
<li class="">Verify and consume nonce. Nonce is an arbitrary number, that has to be unique among all the non-expired signatures of the address.</li>
<li class="">Build the expected <a href="https://github.com/stellar/stellar-xdr/blob/e372df9f677961aac04c5a4cc80a3667f310b29f/Stellar-transaction.x#L703" target="_blank" rel="noopener noreferrer" class="">signature payload preimage</a> and compute its SHA-256 hash to get the final signature payload</li>
<li class="">Call <code>__check_auth</code> of the account contract corresponding to the <code>Address</code> using the signature payload and the invocations from the authorization tree</li>
</ul>
</li>
<li class="">Mark the invocation as &#x27;exhausted&#x27; in its authorized invocation tree. &#x27;Exhausted&#x27; invocations will be skipped when matching the future <code>require_auth</code> calls.</li>
</ul>
<p>If any of the steps above fails, then the authorization is considered unsuccessful.</p>
<p>Notice, that authentication happens just once per tree, as the whole tree needs to be signed.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="matching-authorized-invocation-trees">Matching Authorized Invocation Trees<a href="#matching-authorized-invocation-trees" class="hash-link" aria-label="Direct link to Matching Authorized Invocation Trees" title="Direct link to Matching Authorized Invocation Trees" translate="no">​</a></h4>
<p>In order for authorizations to succeed, all the <code>require_auth</code>/<code>require_auth_for_args</code> calls have to be covered by the corresponding <code>SorobanAuthorizedInvocation</code> trees in a transaction (defined in transaction <a href="https://github.com/stellar/stellar-xdr/blob/e372df9f677961aac04c5a4cc80a3667f310b29f/Stellar-transaction.x#L537" target="_blank" rel="noopener noreferrer" class="">XDR</a>).</p>
<p>Formally, this correspondence is defined as follows.</p>
<p>Given a top-level contract invocation <code>I</code> we can build a &#x27;contract invocation tree&#x27; <code>T</code> by tracing all the sub-contract calls (a directed edge <code>A-&gt;B</code> in the tree means &#x27;contract function A calls contract function B). Note, that we only consider the functions that are implemented in different contracts, i.e. any function calls that don&#x27;t involve a contract invocation via host <code>call</code> are considered to belong to the same node.</p>
<p>Let&#x27;s say authorization is required from addresses <code>A_1..A_N</code>. Then for every address <code>A_i</code> there are two kinds of nodes in the invocation tree <code>T</code>: <code>R</code>-nodes that had a <code>require_auth</code> call for <code>A_i</code> and <code>N</code>-nodes that didn&#x27;t have such call. Then we remove all the <code>N</code>-nodes and all the edges from <code>T</code> and add the directed edges connecting the remaining <code>R</code>-nodes such that the edge goes from <code>R_j</code> to <code>R_k</code> if there was a path between <code>R_j</code> and <code>R_k</code> in <code>T</code> that doesn&#x27;t contain any other <code>R</code>-nodes. As a result we get a forest of <code>SorobanAuthorizedInvocation</code> trees for <code>A_i</code>. Notice, that these trees don&#x27;t have to have their root be <code>I</code> node (i.e. the top-level contract call), so it&#x27;s possible to e.g. batch the authorized call together without requiring signing the batching function.</p>
<p>In simpler terms, <code>SorobanAuthorizedInvocation</code> trees for an <code>Address</code> are subsets of the full invocation tree that are &#x27;condensed&#x27; to only contain invocations that have <code>require_auth</code> call for that <code>Address</code>.</p>
<p>During the matching process that happens for every <code>require_auth</code> host tries to match the current path in <code>T</code> to a <code>SorobanAuthorizedInvocation</code> tree for the corresponding <code>Address</code>. The path is considered to be matched only when there is a corresponding path of <em>exhausted</em> <code>R</code> nodes leading to the current call. This means that if the <code>Address</code> signs a sequence of calls <code>A.foo-&gt;B.bar-&gt;C.baz</code>, then its authorization check will fail in case if <code>A.foo</code> directly calls <code>C.baz</code> because <code>C.baz</code> strictly has to be called from <code>B.bar</code>.</p>
<h5 class="anchor anchorTargetStickyNavbar_Vzrq" id="duplicate-addresses">Duplicate Addresses<a href="#duplicate-addresses" class="hash-link" aria-label="Direct link to Duplicate Addresses" title="Direct link to Duplicate Addresses" translate="no">​</a></h5>
<p>In case if the same contract function calls <code>require_auth</code> for the same <code>Address</code> multiple times (e.g. when multiple operations from the same user are being batched), every <code>require_auth</code> call still has to have a corresponding node in the <code>SorobanAuthorizedInvocation</code> tree. Due to that, there might be multiple valid trees that make all the authorization checks pass. There is nothing wrong about that - the address still must have authorized all the invocations. The only requirement for such cases to be handled correctly is to ensure that the <code>require_auth</code> calls for an <code>Address</code> happen before the corresponding sub-contract
